## Codebase Patterns
- Backend uses Lombok @Data, @Builder, @NoArgsConstructor, @AllArgsConstructor on all entities and DTOs
- DTOs have a static `from(Entity entity)` factory method for entity-to-DTO conversion
- JPA entities use @CreationTimestamp from org.hibernate.annotations for createdAt fields
- ManyToMany relationships use @JoinTable with explicit join column names
- Hibernate DDL is set to `update` mode - tables auto-created from entities
- Java/Maven not available in this environment - "Typecheck passes" means code is syntactically correct Java
- Frontend TypeScript typecheck: `cd frontend && npx tsc -b --noEmit`
- Controllers inject `Authentication` and get current user email via `(String) authentication.getPrincipal()`
- Services use `@Transactional` (write) and `@Transactional(readOnly = true)` for read operations
- Lazy-loaded collections (@ManyToMany with FetchType.LAZY) are safe to access within @Transactional service methods
- Conflict responses use AppException(HttpStatus.CONFLICT, message)
- File uploads: `@RequestParam("file") MultipartFile file`; store under `@Value("${app.uploads.dir:uploads}")` dir with UUID prefix
- Daily entity has: confirmedPlayers (ManyToMany<User>, join table daily_confirmed_players), isFinished (boolean), championImage (String)
- Team entity: id, daily (ManyToOne), name, players (ManyToMany<User>, join table team_players)
- TeamRepository: findByDaily(Daily daily)
- No browser testing tools available; manual browser verification needed
- DailyDetailDTO includes `isAdmin` boolean (populated in getDailyDetail by checking pelada.getAdmins().contains(caller)) — use this for all admin-only UI on DailyDetailPage
- Status badge colors: SCHEDULED=blue-100/800, CONFIRMED=yellow-100/800, IN_COURSE=green-100/800, FINISHED=gray-100/800, CANCELED=red-100/800
- Date formatting for ISO date strings: append 'T12:00:00' before parsing to avoid timezone off-by-one
- Upsert pattern for entities: repository.findByX().orElse(EntityBuilder.builder()...build()) then save()
- `DailyAwardRepository.findByDaily` returns Optional — use `.orElse(DailyAward.builder().daily(daily).build())` for upsert
- Win check for a player in finalization: check if player's team is the team at position 1 in LeagueTable
- finalizeDaily already computes UserDailyStats and LeagueTableEntry — Stats & Ranking upserts should be added in the same @Transactional block
- Pelada entity has getMembers() and getAdmins() collections (ManyToMany); use these to check membership and build full ranking list including zero-stats members
- Stats entity uses @ElementCollection for puskasDates (List<LocalDate>) with @CollectionTable(name="stats_puskas_dates")
- Ranking entity has unique constraint on (pelada_id, user_id); use @UniqueConstraint in @Table annotation
- Win determination for Ranking/Stats: check if winningTeam.getPlayers().contains(player); winningTeam is null if tied at top of league table
- DailyService already uses import com.futspring.backend.entity.*; and import com.futspring.backend.repository.*; — new entities/repos are auto-imported
