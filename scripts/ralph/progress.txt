## Codebase Patterns
- When extracting a service, the delegation stub in the original service must @Transactional-annotate wrapper methods and call the sub-service; the original service's tests become delegation tests (mock sub-service → verify call), and a new test file covers the real logic
- `DailyAttendanceService` is in the same package as `DailyService` so clearAttendees (package-private) is accessible to future orchestration code in that package
- Backend services use `@Service` + `@RequiredArgsConstructor` + `private final` constructor injection (no @Autowired field injection)
- AppException(HttpStatus, message) is the standard error pattern
- Tests use `new ServiceClass(repo1, repo2, ...)` with Mockito `@Mock` fields — no Spring context
- When adding a new service dependency, update ALL test files that `new` the affected service class
- `@Value("${app.uploads.dir:uploads}")` fields cannot coexist with `@RequiredArgsConstructor` if you want them injected — use `@Value` on a non-final field instead, or move to a config bean
- File upload logic uses: 5MB size check → MIME check (jpeg/png/webp) → UUID filename → Files.copy → return filename; IOException → AppException(500, "Failed to store file")
- UserAuthenticationHelper lives in `com.futspring.backend.helper` and delegates user lookup to avoid repeating the findByEmail pattern; services that removed their userRepository dependency (ChatService, RankingService, AwardsService) drop the repo from their imports but DailyService/PeladaService retain it for other uses
- Tests that check user-not-found now mock `userAuthHelper.getAuthenticatedUser(email)` to throw `AppException(NOT_FOUND)` instead of returning `Optional.empty()` from userRepository — and assert NOT_FOUND (not UNAUTHORIZED)
- `DailyDTOMapper` is a `@Component` (not `@Service`) that holds `buildTeamDTO` — injected into both `DailyTeamManagementService` and `DailyService` (for `getDailyDetail`); use `dailyDTOMapper::buildTeamDTO` method reference, never `this::buildTeamDTO`
- `DailyTeamManagementService` and `DailyDTOMapper` live in the same `service` package as `DailyService`; `clearTeams` is package-private (no visibility modifier)
